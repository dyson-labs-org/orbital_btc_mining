## Purpose

Evaluate whether the Cycle 2 harness remediations work in ordinary external
repository use. The agent-engineering-harness is the product under evaluation;
Orbital Compute Lab is the controlled incubation vessel.

## Product Vessel Task

Add `scenario-suite.v1`, a deterministic runner that executes existing resource
transition scenarios and compares actual domain outcomes with declared expected
outcomes. The task must not add scheduling, workloads, optimization, providers,
adapters, network behavior, hardware control, mission authority, or dependencies.

## Cycle 2 Remediation Claims

Cycle 3 tests these harness claims from the merged remediation branch:

- artifact references can be rehashed by a first-class integrity command;
- external-repository evidence can be generated without bespoke glue;
- review packets can be bundled with structured supplements;
- new persisted evidence uses logical paths instead of raw machine-local paths.

## Evaluation Questions

1. Does an installed harness at the frozen SHA generate Orbital evidence outside
   both source checkouts without `PYTHONPATH`, editable installation, or custom
   evidence-generation scripts?
2. Does integrity validation detect tampered, missing, stale, dirty, and
   relocated evidence without claiming authenticity?
3. Are persisted references logical and free of raw local absolute paths?
4. Can review bundles validate required supplements, source commits, duplicate
   IDs, and referenced bytes?
5. Can a fresh reviewer evaluate the product and harness evidence without this
   full milestone prompt?
6. Have the Cycle 2 findings been closed, partially closed, or left open in
   normal external-repository use?

## Confounders

- The local sandbox did not permit creating a root-level temporary directory, so
  external artifacts may use the process temporary directory while still
  remaining outside both repositories.
- Static source scans are not operating-system syscall isolation.
- Runtime evidence is ignored and must be summarized in committed documents
  without copying machine-local paths into the repository.
- The installed harness method is the repository's no-network installed-style
  smoke procedure, not a package publish or package-manager install.

## Stop Conditions

Stop if either prerequisite PR is not merged, either main is dirty or
unsynchronized before freezing, the legacy branch pin differs, Node.js 22 or
newer is unavailable, evidence generation requires bespoke glue, raw local paths
persist in required artifacts, tampered or missing required artifacts validate as
intact, stale evidence is represented as current, a required supplement can be
silently omitted, target mutation is hidden, or a critical/high harness finding
cannot be honestly dispositioned.

## Conclusion Rules

- `pass`: all four Cycle 2 remediation areas work in ordinary external-repository
  use, every integrity and path-hygiene gate passes, ordinary evidence generation
  needs no custom glue, and no unresolved critical/high harness finding remains.
- `pass_with_findings`: integrity and safety gates pass, with remaining
  noncritical usability, documentation, or portability findings supported by
  evidence and dispositions.
- `fail`: raw machine-local paths or synthetic secrets persist, tampered or
  missing artifacts validate as intact, stale artifacts are represented as
  current, ordinary evidence requires bespoke glue, required supplements can be
  omitted silently, review output cannot be tied to the reviewed commit, or an
  unresolved critical/high harness issue remains.
